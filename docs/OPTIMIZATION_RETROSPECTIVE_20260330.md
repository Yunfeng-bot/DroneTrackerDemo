# 追踪优化复盘（2026-03-30）

## 1. 目标与范围

本轮工作的目标是两件事：

1. 让“同一目标图 + 同一视频”下的评估结果可复现，避免随机波动误导优化方向。
2. 在不牺牲稳定锁定的前提下，继续压低首锁耗时与平均帧耗时。

当前复盘仅覆盖 Android 端 App 的追踪主链路，不涉及 Linux/NDK 新架构迁移。

## 2. 本轮优化点（表格）

| 优化项 | 解决的问题 | 具体改动 | 实际效果（通俗描述） |
|---|---|---|---|
| 固定视频回放评测链路 | 现场相机画面波动大，调参容易“看起来有效，下一轮又失效” | 增加 replay 输入模式，统一同源目标图和视频 | 从“看运气调参”变成“同题同解对比”，结论更可信 |
| 评测日志结构化 | 空日志/不完整日志导致无法判断失败原因 | 增加 `EVAL_EVENT/EVAL_PERF/EVAL_SUMMARY` 抽取与报告 | 出问题时能快速定位是“没启动”“没锁到”还是“速度过慢” |
| Enhanced 轻量化 | 精修太频繁导致热负载上升 | 调整局部精修频率和局部搜索范围策略 | 追踪稳定前提下，平均耗时略降 |
| 运行时热调参 | 每次改参数都要改代码重装，迭代很慢 | 增加 `eval_params` 运行时参数注入入口 | 一版 APK 可反复试参数，迭代效率明显提升 |
| 启动与重入稳定性 | Intent 复用时参数可能不生效 | 增加 `onNewIntent` 重配置流程 | 自动化连续评测时“参数未生效”问题显著减少 |

## 3. 验证结果汇总（同一目标图+同一视频）

| 指标 | Baseline（对照） | Enhanced（优化后） |
|---|---:|---:|
| 首锁时间（中位数） | 未稳定锁定 | 约 1.99s |
| 平均帧耗时（中位数） | 约 1044ms | 约 71ms |
| 锁定成功率 | 接近 0 | 1.00（多轮稳定锁定） |
| 追踪占比 | 接近 0 | 1.00 |

结论：当前阶段应以 Enhanced 方案作为主线，Baseline 仅保留为参考对照。

## 4. 踩坑复盘（避免重复）

| 坑点 | 现象 | 规避规则 |
|---|---|---|
| 并行在同一设备跑多组评测 | 日志互相覆盖、结论失真 | 单设备严格串行执行评测任务 |
| 日志目录命名粒度太粗 | 同秒运行覆盖前一轮结果 | 运行目录使用毫秒级时间戳 |
| 参数只在首次启动生效 | 连续评测后参数“看似没改” | 统一走 Intent 重入配置路径 |
| 只看单次结果就定参数 | 容易被偶发波动带偏 | 默认看多轮中位数，不看单次极值 |

## 5. 清理结论

按当前要求，自动化验证工具与产物已从项目中清理，仅保留 App 自身代码链路用于后续开发与交付。
